Как это работает

Логика устройства от запроса до экрана

Шаги при каждом обновлении (раз в 2–3 минуты)

  • 1
    Запрос sensorsNearby

    Отправляются координаты, радиус поиска и коды нужных типов датчиков. Сервер возвращает список устройств с их датчиками в радиусе.

  • 2
    Фильтр по свежести

    Из ответа отбрасываются датчики, у которых time старше 15 минут — они считаются неактивными.

  • 3
    Группировка по типу

    Все датчики раскладываются по 8 категориям: температура, влажность, давление, ветер (2 типа), радиация, пыль, осадки.

  • 4
    Сортировка по расстоянию, отбор N ближайших

    Внутри каждой категории датчики сортируются по полю distance, берутся первые N (настраиваемо, по умолчанию 5).

  • 5
    Взвешенное усреднение (IDW)

    Чем ближе датчик — тем больше его вклад в итоговое значение. Формула в разделе 2.

  • 6
    Проверка тревог

    Если итоговое значение радиации/ветра/температуры/пыли превышает порог — включается тревожный режим.

💡 Почему не угадываем коды типов датчиков: народмон не публикует таблицу числовых кодов type открыто. Прошивка сама запрашивает справочник через метод appInit при старте и находит нужные коды по точному названию типа на русском — это надёжнее, чем хардкодить числа, которые могут не совпасть.
⚠ Лимит API: народмон разрешает не чаще 1 запроса в минуту с одного ключа, иначе HTTP 429 и возможна блокировка. В прошивке интервал обновления данных — 180 секунд (3 минуты), что оставляет большой запас.

Формула усреднения IDW

Inverse Distance Weighting — чем ближе, тем больше вес

Формула

wi = 1 / (di + ε)p

x̄ = Σ(wi · xi) / Σwi

ПараметрСмыслВ прошивке
diрасстояние до датчика i, кмиз поля distance ответа API
pстепень затухания — выше p, резче падает влияние дальнихIDW_POWER, по умолчанию 1.0
εзащита от деления на 0 при d→0IDW_EPS = 0.1 км
Nсколько ближайших датчиков берёмNEAREST_COUNT, по умолчанию 5

Пример расчёта (p=1, датчики на 1, 3 и 10 км)

ДатчикРасстояниеВес wДоля влияния
11 км1/1.1 ≈ 0.90966%
23 км1/3.1 ≈ 0.32323%
310 км1/10.1 ≈ 0.0997%

При p=1 ближний датчик доминирует, но дальние всё равно вносят заметный вклад — это разумно для погодных параметров, которые меняются плавно в пространстве.

📌 Все четыре параметра (NEAREST_COUNT, IDW_POWER, IDW_EPS, координаты MY_LAT/MY_LON) — обычные переменные в начале прошивки, не константы. Можно менять без понимания остального кода.

Шаг 3 — настроить TFT_eSPI (ОБЯЗАТЕЛЬНО, иначе экран не заработает)

Библиотека TFT_eSPI не определяет дисплей автоматически — нужно вручную прописать модель и пины в файле настроек.

  • 1
    Найти папку библиотеки

    Обычно: Documents/Arduino/libraries/TFT_eSPI/

  • 2
    Открыть файл User_Setup.h

    Закомментировать (поставить //) всё содержимое и заменить на конфигурацию ниже.

  • 3
    Вставить конфигурацию для ST7789 240×320
    User_Setup.h — добавить в конец файла
    #define ST7789_DRIVER
#define TFT_WIDTH  240
#define TFT_HEIGHT 320

#define TFT_MOSI 23
#define TFT_SCLK 18
#define TFT_CS   15
#define TFT_DC   2
#define TFT_RST  4

#define LOAD_GLCD
#define LOAD_FONT2
#define LOAD_FONT4
#define LOAD_FONT6
#define LOAD_FONT7
#define LOAD_FONT8
#define LOAD_GFXFF

#define SPI_FREQUENCY  40000000
  • 4
    Сохранить файл, перезапустить Arduino IDE

    Изменения в библиотечных файлах требуют перезапуска IDE.

💡 Если экран остаётся белым/чёрным после загрузки — почти всегда проблема в неверных пинах TFT_CS/DC/RST в User_Setup.h. Проверьте их соответствие реальной распиновке вашего модуля (бывают разные варианты у разных продавцов).

Поведение при тревоге

  • 1
    Карусель экранов

    Через раз показывается тревожный экран (температура крупно + красный баннер с названием тревоги) вместо обычной карусели.

  • 2
    Светодиод

    Мигает постоянно (раз в 400 мс) пока хотя бы одна тревога активна.

  • 3
    Пищалка

    Короткий сигнал (300 мс, 2000 Гц) раз в 15 минут — не каждую минуту, чтобы не раздражать.

  • 4
    Автосброс

    Как только все параметры возвращаются в норму при следующем обновлении данных — тревога снимается автоматически.

Прошивка целиком

Файл weather_display.ino — основная логика

Полный код прошивки (632 строки) приложен отдельным файлом weather_display.ino. Основные блоки:

ФункцияЧто делает
loadSensorTypeCodes()Запрашивает appInit, находит коды типов по названию
fetchAndAverage()Запрашивает sensorsNearby, фильтрует свежесть, считает IDW
checkAlarms()Сравнивает результаты с порогами тревог
handleAlarmIndication()Управляет пищалкой и светодиодом
drawSensorScreen()Рисует один экран карусели на TFT
updateCarousel()Решает какой экран показать в данный момент
📌 Структура sensorTypes[] в начале файла содержит точные названия типов датчиков на русском, по которым ищется код в справочнике appInit. Если народмон вернёт другое название (например регистр отличается) — в Serial Monitor появится предупреждение "тип не найден", и можно скорректировать строку поиска.
